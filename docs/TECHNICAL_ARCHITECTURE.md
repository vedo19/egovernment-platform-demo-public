**Key architectural decisions:**
- Every microservice owns its database — no service directly accesses another service's database
- All external traffic enters through the API Gateway — clients never call services directly
- JWT tokens are generated by the Auth Service and validated independently by each service using a shared secret key
- Services are stateless — all state lives in the databases

## Microservice Design Principles

### 1. Database-per-Service
Each microservice has its **own PostgreSQL database** running in its own Docker container. This ensures:
- Services can evolve their schema independently
- No tight coupling through shared tables
- Each database can be scaled, backed up, or migrated independently

### 2. Independent Deployability
Each service has its own `Dockerfile`, its own `.csproj`, and its own `Program.cs`. You can rebuild and redeploy one service without touching the others.

### 3. API-First Communication
Services do not call each other directly. Each service exposes a REST API and the frontend (or other services) communicate through HTTP calls via the API Gateway.

**Current inter-service link:** The Citizen Service and Service Request Service both store `CitizenUserId` (the user's GUID from the Auth Service). This is a data reference, not a runtime dependency — each service stores the user ID when the request is created.

### 4. Shared Authentication
All services share the same JWT configuration (secret key, issuer, audience). When the Auth Service issues a JWT token, any other service can validate it independently without calling back to the Auth Service. The JWT contains:
- `sub` — User ID (GUID)
- `email` — User email
- `name` — User full name
- `role` — User role (Citizen, Admin, Officer)
- `jti` — Unique token identifier
- `exp` — Expiration timestamp

### 5. Controller-Based (MVC Pattern)
All services use ASP.NET Core MVC Controllers (not Minimal APIs). Each controller handles routing, input validation, authorization, and delegates business logic to a service layer.

```
Controller → Service Interface → Service Implementation → EF Core DbContext → PostgreSQL
```

## Service-by-Service Breakdown

### Auth Service (port 5001)

**Purpose:** User registration, authentication, and JWT token generation.

**Internal architecture:**
```
AuthController
  ├── POST /register       → AuthServiceImpl.RegisterAsync()
  ├── POST /login          → AuthServiceImpl.LoginAsync()
  ├── GET  /me             → AuthServiceImpl.GetUserByIdAsync()
  ├── GET  /users          → AuthServiceImpl.GetAllUsersAsync()
  └── PUT  /users/{id}/role → AuthServiceImpl.UpdateUserRoleAsync()  [Admin only]
                              │
                      TokenService.GenerateToken()
                              │
                      AuthDbContext → Users table
```

**Key implementation details:**
- Passwords are hashed with **BCrypt** (via BCrypt.Net-Next) — never stored in plain text
- Email addresses are **normalized** (trimmed + lowered) before storage to prevent duplicate accounts
- JWT tokens are signed with **HMAC-SHA256** using a shared secret key
- Token expiry: 60 minutes
- `MapInboundClaims = false` on all services — this prevents ASP.NET from remapping JWT claim names (e.g., it keeps "sub" as "sub" instead of remapping to a long URI)
- **Registration is locked to Citizen role** — the `role` field in the request body is ignored; all new users are created as Citizens
- **Admin seeding on startup** — the Auth Service checks for `Admin:Email` / `Admin:Password` / `Admin:FullName` configuration values (set via environment variables `Admin__Email`, etc.) and creates an Admin user on first boot if one doesn't exist. Use environment-specific values and never publish real credentials.
- **Role management** — the `PUT /users/{id}/role` endpoint (Admin only) allows promoting/demoting users between Citizen, Officer, and Admin roles

**Database:** `auth_db` on port 5436 (external)
- Table: `Users` — Id (PK, UUID), FullName, Email (unique), PasswordHash, Role, CreatedAt, UpdatedAt

---

### Citizen Service (port 5002)

**Purpose:** CRUD operations for citizen profiles, linked to Auth Service users by UserId.

**Internal architecture:**
```
CitizensController
  ├── POST /profile       → CitizenServiceImpl.CreateProfileAsync()
  ├── GET  /profile       → CitizenServiceImpl.GetProfileByUserIdAsync()
  ├── PUT  /profile       → CitizenServiceImpl.UpdateProfileAsync()
  ├── GET  /              → CitizenServiceImpl.GetAllProfilesAsync()
  ├── GET  /{id}          → CitizenServiceImpl.GetProfileByIdAsync()
  └── GET  /by-user/{uid} → CitizenServiceImpl.GetProfileByUserIdAsync()
                                    │
                            CitizenDbContext → CitizenProfiles table
```

**Key details:**
- Each citizen has exactly one profile (enforced by unique constraint on UserId)
- NationalId is validated for uniqueness — no two citizens can have the same national ID
- DateOfBirth uses `DateOnly` (not DateTime) — avoids PostgreSQL UTC timezone complications
- Profile data comes from two sources: personal info from the form + FullName/Email from the JWT claims

**Database:** `citizen_db` on port 5433
- Table: `CitizenProfiles` — Id, UserId (unique), FullName, Email, PhoneNumber, NationalId (unique), DateOfBirth, Address, City, Gender, CreatedAt, UpdatedAt

---

### Service Request Service (port 5003)

**Purpose:** Citizens submit service requests (permits or complaints), and admins/officers manage them.

**Internal architecture:**
```
ServiceRequestsController
  ├── POST /              → Create new request (Citizen)
  ├── GET  /my-requests   → Get citizen's own requests
   ├── GET  /my-assignments → Get requests assigned to current officer (Officer only, legacy)
   ├── GET  /assigned-to-me → Get requests assigned to current officer (Officer)
   ├── GET  /all           → List all requests (Admin)
  ├── GET  /              → List all requests with optional ?status= filter (Admin/Officer)
  ├── GET  /{id}          → Get single request (owner check for Citizens)
   ├── PUT  /{id}/status   → Update status (Admin/Officer, legacy compatibility)
   ├── PUT  /{id}/assign   → Assign officer (Admin only, legacy)
   ├── PUT  /{id}/assign-officer → Assign officer (Admin)
   ├── PUT  /{id}/request-documents → Request permit PDF (Admin/Officer)
   ├── PUT  /{id}/approve  → Approve request (Admin/Officer)
   ├── PUT  /{id}/reject-documents → Reject permit document only (Admin/Officer)
   ├── PUT  /{id}/reject   → Final rejection (Admin/Officer)
   └── POST /{id}/upload-document → Citizen uploads PDF (multipart/form-data)
                                    │
                            ServiceRequestDbContext → ServiceRequests table
```

**Workflow status flow:**

Permit:

```
Submitted → OfficerAssigned → AwaitingDocuments → UnderReview → Approved
                                                             ↘ citizen upload ↗
UnderReview → DocumentsRejected → UnderReview
UnderReview → Rejected (final)
```

Complaint:

```
Submitted → OfficerAssigned → UnderReview → Approved
                                                          ↘ Rejected (final)
```

**Key details:**
- `Type` field: "Permit" or "Complaint"
- Citizens can only view their own requests (enforced in controller by comparing CitizenUserId with JWT sub claim)
- Progress is computed on read: `ProgressPercentage` + `ProgressColor`
- Permit document resubmission is supported via `DocumentsRejected`
- Request now stores `OfficerNote` and `LinkedDocumentId`
- `ResolvedAt` is automatically set when status reaches final states
- ServiceRequest orchestrates PDF storage through DocumentService internal API call

**Database:** `request_db` on port 5434
- Table: `ServiceRequests` — Id, CitizenUserId, Type, Title, Description, Status, AdminNotes, OfficerNote, AssignedOfficerId, LinkedDocumentId, CreatedAt, UpdatedAt, ResolvedAt

### Document Service (port 5004)

**Purpose:** Citizens request official documents; admins/officers process them and generate reference numbers. Also stores permit supporting PDFs uploaded from ServiceRequest workflow.

**Internal architecture:**
```
DocumentsController
  ├── POST /              → Create document request (Citizen)
  ├── GET  /my-documents  → Get citizen's own documents
   ├── GET  /my-assignments → Get documents assigned to current officer (Officer only, legacy)
   ├── GET  /assigned-to-me → Get documents assigned to current officer (Officer)
  ├── GET  /              → List all with ?status=&documentType= filter (Admin/Officer)
  ├── GET  /{id}          → Get single document (owner check for Citizens)
   ├── PUT  /{id}/status   → Update status (Admin/Officer, legacy compatibility)
   ├── PUT  /{id}/start-review → Move Submitted → UnderReview (Admin/Officer)
   ├── PUT  /{id}/approve  → Move UnderReview → Approved (Admin/Officer)
   ├── PUT  /{id}/reject   → Move UnderReview → Rejected (Admin/Officer)
   ├── PUT  /{id}/assign   → Assign officer (Admin only)
   ├── POST /supporting-files → Upload supporting PDF into DB (Citizen/internal)
   └── GET  /supporting-files/{id}/download → Download supporting PDF
                                    │
                                          DocumentDbContext → Documents + SupportingDocuments tables
```

**Workflow status flow:**

```
Submitted → UnderReview → Approved
                               ↘ Rejected
```

**Key details:**
- Document types: BirthCertificate, NationalId, MarriageCertificate, DeathCertificate, DrivingLicense
- **Reference number generation:** When a document status is changed to "Approved", a unique reference number is automatically generated in the format: `{TypePrefix}-{Date}-{RandomHex}` (e.g., `NI-20260306-ABC12345`)
- Type prefixes: BC (Birth Certificate), NI (National ID), MC (Marriage Certificate), DC (Death Certificate), DL (Driving License)
- Reference numbers have a unique index with null filter (only non-null values are unique)
- `RejectionReason` is stored when status is set to Rejected
- Progress is computed on read: `ProgressPercentage` + `ProgressColor`
- Supporting permit PDFs are stored in `SupportingDocuments` with bytea content and metadata

**Database:** `document_db` on port 5435
- Table: `Documents` — Id, CitizenUserId, DocumentType, Title, Description, Status, RejectionReason, AssignedOfficerId, ProcessedByOfficerId, ReferenceNumber (unique filtered), CreatedAt, UpdatedAt, CompletedAt
- Table: `SupportingDocuments` — Id, CitizenUserId, ServiceRequestId, FileName, ContentType, FileData, FileSize, UploadedAt

## Database Architecture

### Isolation Strategy

Each service uses a completely separate PostgreSQL instance:

| Service | Database Name | Container | External Port | Internal Port |
|---------|--------------|-----------|---------------|---------------|
| Auth | auth_db | auth_db | 5436 | 5432 |
| Citizen | citizen_db | citizen_db | 5433 | 5432 |
| Service Request | request_db | request_db | 5434 | 5432 |
| Document | document_db | document_db | 5435 | 5432 |

All databases use:
- **PostgreSQL 16 Alpine** (lightweight Docker image)
- **Credentials:** provided via environment variables (`POSTGRES_USER`, `POSTGRES_PASSWORD`) per environment
- **Persistent volumes:** data survives container restarts
- **Health checks:** `pg_isready` every 5 seconds — services wait until their DB is healthy before starting

### ORM & Migrations

- **Entity Framework Core 10.x** with **Npgsql 10.0.0** (PostgreSQL provider)
- **Auto-migration on startup:** Each service's `Program.cs` calls `context.Database.Migrate()` at startup — tables are created/updated automatically
- No manual SQL scripts needed — EF Core handles schema creation

## Authentication & Authorization Flow

### Admin Seeding (on startup)
```
1. Auth Service starts up and runs migrations
2. Reads Admin:Email, Admin:Password, Admin:FullName from configuration
3. Checks if a user with that email already exists
4. If not → creates a new User with Role = "Admin"
5. Admin can now log in and promote other users
```

### Registration Flow
```
1. Client → POST /api/auth/register {fullName, email, password}
2. Auth Service:
   a. Normalizes email (trim + lowercase)
   b. Checks for existing email
   c. Hashes password with BCrypt
   d. Saves User with Role = "Citizen" (hardcoded, ignores any role in request)
   e. Generates JWT token with claims (sub, email, name, role, jti)
3. Client ← {userId, email, fullName, role: "Citizen", token}
4. Client stores token in localStorage
```

### Login Flow
```
1. Client → POST /api/auth/login {email, password}
2. Auth Service:
   a. Finds user by normalized email
   b. Verifies password with BCrypt
   c. Generates new JWT token
3. Client ← {userId, email, fullName, role, token}
4. Client stores token in localStorage
```

### Authenticated Request Flow
```
1. Client → GET /api/citizens/profile
   Headers: Authorization: Bearer <jwt-token>

2. API Gateway (Ocelot):
   a. Receives request on port 5050
   b. Matches route /api/citizens/* → citizen_service:80
   c. Forwards request with all headers (including Authorization)

3. Citizen Service:
   a. ASP.NET JWT middleware validates the token:
      - Verifies HMAC-SHA256 signature using shared secret key
      - Checks issuer = "AuthService"
      - Checks audience = "EGovernmentPlatform"
      - Checks expiration (ClockSkew = Zero)
   b. Extracts claims: sub → userId, role → "Citizen"
   c. [Authorize(Roles = "Citizen")] checks the role claim
   d. Controller reads userId from "sub" claim
   e. Queries database for profile matching userId

4. Client ← {profile data} or 404 if not found
```

### JWT Token Structure (decoded)
```json
{
  "sub": "a0730905-5e40-40b1-8c5c-b2289efea0e1",
  "email": "citizen@example.com",
  "name": "John Doe",
  "role": "Citizen",
  "jti": "d482960b-90ab-4857-9799-8fb47cfd1ffb",
  "exp": 1772836988,
  "iss": "AuthService",
  "aud": "EGovernmentPlatform"
}
```

### Role-Based Access Control (RBAC)

| Endpoint Pattern | Citizen | Officer | Admin |
|-----------------|---------|---------|-------|
| Register / Login | Yes | Yes | Yes |
| Own profile CRUD | Yes | No | No |
| Submit requests/documents | Yes | No | No |
| View own requests/documents | Yes | No | No |
| View assigned tasks (my-assignments) | No | Yes | No |
| Review & approve/reject assigned tasks | No | Yes | No |
| View ALL requests/documents | No | No | Yes |
| Update request/document status | No | No | Yes |
| Assign officer to request | No | No | Yes |
| Change user roles | No | No | Yes |
| List all users | No | No | Yes |

## API Gateway

### Technology: Ocelot (v24.1.0)

Ocelot is an open-source API Gateway built for .NET. It sits in front of all microservices and handles:

1. **Routing** — Maps incoming URLs to downstream services
2. **Header forwarding** — Passes through all headers, including `Authorization: Bearer <token>`
3. **Single entry point** — Clients only need to know about port 5050

### Route Configuration (ocelot.json)

Each route has:
- `UpstreamPathTemplate` — the URL the client calls (e.g., `/api/auth/{everything}`)
- `DownstreamPathTemplate` — the URL the service expects (same path)
- `DownstreamHostAndPorts` — Docker internal hostname and port (e.g., `auth_service:80`)

### Why an API Gateway?

Without the gateway, the frontend would need to know about 4 different service URLs:
```
http://localhost:5001/api/auth/...
http://localhost:5002/api/citizens/...
http://localhost:5003/api/servicerequests/...
http://localhost:5004/api/documents/...
```

With the gateway, everything goes through one URL:
```
http://localhost:5050/api/...
```

This is essential for:
- Simplified frontend configuration
- Future load balancing
- Centralized logging and monitoring
- Future rate limiting and request throttling

## Frontend Architecture

### Technology: React 19 + Vite 7 + React Router 7

The frontend is a **Single Page Application (SPA)** — a single HTML page that dynamically renders different views based on the URL.

### Structure

```
src/
├── api/
│   ├── client.js         ← Axios HTTP client (base URL, JWT interceptor)
│   └── services.js       ← API functions grouped by service
├── context/
│   └── AuthContext.jsx    ← Global auth state (user, login, register, logout)
├── components/
│   ├── Layout.jsx         ← Navbar + content area (uses React Router <Outlet>)
│   └── ProtectedRoute.jsx ← Route guard (redirects to /login if not authenticated)
└── pages/
    ├── LoginPage.jsx        ← Email/password form
    ├── RegisterPage.jsx     ← Registration (always creates Citizen)
    ├── CitizenDashboard.jsx ← 3-tab dashboard (Profile, Requests, Documents)
    ├── AdminDashboard.jsx   ← 3-tab dashboard (Requests, Documents, Users + role management)
    └── OfficerDashboard.jsx ← 2-tab dashboard (My Requests, My Documents) — assigned tasks only
```

### Key Patterns

**Auth interceptor (client.js):**
- Every outgoing request automatically attaches `Authorization: Bearer <token>` from localStorage
- If any response returns 401 (Unauthorized), the token is cleared and the user is redirected to `/login`

**Auth context (AuthContext.jsx):**
- React Context provides `user`, `login()`, `register()`, `logout()` to all components
- User data is persisted in localStorage so it survives page refreshes
- On login/register, the JWT token and user info are stored simultaneously

**Role-based routing (App.jsx):**
```
/login          → LoginPage (public)
/register       → RegisterPage (public)
/citizen        → CitizenDashboard (requires role: Citizen)
/officer        → OfficerDashboard (requires role: Officer)
/admin          → AdminDashboard (requires role: Admin)
/  (root)       → Redirects to /citizen, /officer, or /admin based on role
```

### Production Deployment

In Docker, the React app is:
1. Built with `npm run build` (produces static files in `dist/`)
2. Served by **nginx** (lightweight web server)
3. nginx config: all routes serve `index.html` (SPA routing via `try_files`)

## Docker & Container Topology

### Container Overview

```yaml
10 containers total:
┌──────────────────────────────────────────────────┐
│  frontend          (nginx, port 3000)             │
├──────────────────────────────────────────────────┤
│  api_gateway       (Ocelot, port 5050)           │
├──────────┬──────────┬──────────┬─────────────────┤
│auth_svc  │citizen   │request   │document          │
│  :5001   │ _svc     │ _svc     │ _svc             │
│          │  :5002   │  :5003   │  :5004           │
├──────────┼──────────┼──────────┼─────────────────┤
│auth_db   │citizen   │request   │document          │
│  :5436   │ _db      │ _db      │ _db              │
│          │  :5433   │  :5434   │  :5435           │
└──────────┴──────────┴──────────┴─────────────────┘
```

### Docker Compose Orchestration

- **Health checks:** Database containers expose health checks using `pg_isready`. Service containers use `depends_on` with `condition: service_healthy` to wait until their database is ready.
- **Named volumes:** Each database uses a named Docker volume (e.g., `auth_db_data`) for persistent storage. Data survives `docker compose down` but is removed with `docker compose down -v`.
- **Internal networking:** All containers are on the same Docker Compose default network. Services reference each other by container name (e.g., `auth_service`, `citizen_db`).

### Dockerfile Pattern (Backend Services)

Each .NET service uses a multi-stage build:
```
Stage 1 (build): SDK image → restore → build → publish
Stage 2 (runtime): ASP.NET runtime image → copy published output → run
```

This keeps the final image small (~220MB vs ~1.5GB for the full SDK).

### Dockerfile Pattern (Frontend)

Two-stage build:
```
Stage 1 (build): Node 22 Alpine → npm ci → npm run build
Stage 2 (serve): nginx Alpine → copy dist/ → serve static files
```

Final image is ~25MB.

## Request Flow — End-to-End Example

Here's what happens when a citizen submits a permit request from the browser:

```
1. User fills out the "New Service Request" form and clicks Submit

2. React (CitizenDashboard.jsx)
   → Calls serviceRequestApi.create({type: 'Permit', title: '...', description: '...'})

3. Axios (client.js)
   → POST http://localhost:5050/api/servicerequests
   → Attaches header: Authorization: Bearer eyJhbGci...

4. API Gateway (Ocelot on port 5050)
   → Matches route: /api/servicerequests → service_request_service:80
   → Forwards the full request with all headers

5. Service Request Service (ASP.NET on port 80 inside container)
   → JWT middleware validates token (signature, expiry, issuer, audience)
   → [Authorize(Roles = "Citizen")] passes (role claim = "Citizen")
   → Controller.Create() extracts userId from "sub" claim
   → ServiceRequestService.CreateAsync(userId, dto)
   → Creates ServiceRequest entity with Status = "Pending"
   → Saves to request_db via EF Core

6. Response flows back:
   Service → Gateway → Axios → React state update → UI re-renders

7. The new request appears in the table with status "Pending" (yellow badge)
```
